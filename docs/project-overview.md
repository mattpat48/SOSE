# Tourism DaaS + EaaS Project Overview

This project is a small service-oriented tourism application that combines a data service and an ethics service around a common travel recommendation flow. It was designed as an academic example of how a domain can be split into separate responsibilities while still delivering a coherent user experience, but also with a specific motivation: to provide ethical support for choosing places to visit according to appropriate criteria such as accessibility, sustainability, and provenance. The overall idea is simple: the system stores and exposes tourism data, evaluates that data against ethical policies, and presents the result through a web client.

At the center of the architecture there are two Spring Boot backends. The DaaS backend, or Data as a Service layer, is responsible for reading and querying the RDF tourism dataset. It exposes REST endpoints that let clients browse places, search by location or category, retrieve detailed information for a specific place, and request recommendation-oriented searches with additional constraints such as accessibility, sustainability, or minimum rating. The EaaS backend, or Ethics as a Service layer, receives a candidate recommendation and evaluates it against a set of policy rules loaded from JSON files. This separation keeps data access and ethical judgment independent, which makes the system easier to understand, extend, and discuss from an architectural point of view.

The front end is built with Vue 3 and Vite and acts as the interaction layer for the user. It offers the main search form, supports browsing results by location, category, and rating, and lets the user inspect the details of a single place before asking for an ethical evaluation. It also includes a dedicated component that displays the decision returned by the EaaS service. In practice, the user starts by searching for tourism options from the interface. The frontend sends a request to the DaaS service, receives matching places, and then allows the user to inspect one of the candidates in more detail. If an ethical evaluation is needed, the selected place is forwarded to the EaaS service, which responds with a decision, a risk level, a rationale, and an audit trail. The final result is displayed back in the browser in a way that makes the recommendation and the reasoning behind it visible at the same time.

The system also uses a shared DTO module to keep the contracts between components aligned. This module contains the data transfer objects used by both backends, such as the request and response payloads for ethical evaluation and the place representation exchanged with the frontend. Keeping these objects in a dedicated shared package avoids duplicated model definitions and reduces the risk of inconsistencies across services.

The data layer is based on an RDF dataset stored under `dataset/` and packaged into the DaaS application resources. The dataset contains a curated set of tourism places in Venice, each described with metadata such as category, location, accessibility, sustainability, provenance, and rating. The DaaS service loads this dataset and uses SPARQL queries through Apache Jena to answer the search requests. This gives the project a semantic data backbone instead of a purely relational or in-memory object store, which is useful for demonstrating how structured knowledge can be queried in a flexible way.

The ethical layer is driven by policy files stored in the EaaS resources folder. Each JSON file describes one policy domain, such as accessibility, sustainability, or provenance. The policy evaluator reads these files at startup and applies them when a recommendation request arrives. The response reflects the outcome of the policy checks and aggregates the resulting risk into a single decision. In this way, the application shows not only what place is recommended, but also why the system considers that place acceptable, risky, or unsuitable.

From an execution standpoint, the project can be run as a full stack application or as individual components. The `run-all.sh` script starts the backends and the frontend together, while the README also documents how to launch each service separately. This makes the project suitable both for demonstrations and for isolated testing of the DaaS or EaaS layer. The repository also includes documentation for the dataset schema, the REST endpoints, and example demo scenarios, which helps explain the system without requiring direct inspection of the code.

Overall, the project is intentionally compact but still representative of a layered service architecture. The DaaS service focuses on facts, the EaaS service focuses on governance and policy, and the frontend ties both together into a user-facing tourism recommendation experience. That division of responsibilities is the main architectural idea behind the repository, and it is what makes the application a clear example of data service and ethics service integration in a real domain.
